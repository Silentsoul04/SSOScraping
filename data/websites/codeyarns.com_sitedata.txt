class=\"code\"><div class=\"container\"><div class=\"line number1 index0 alt2\"><code class=\"plain plain\">$ mkdir build</code></div><div class=\"line number2 index1 alt1\"><code class=\"plain plain\">$ cd build</code></div><div class=\"line number3 index2 alt2\"><code class=\"plain plain\">$ cmake ..</code></div><div class=\"line number4 index3 alt1\"><code class=\"plain plain\">$ make</code></div></div></td></tr></tbody></table></div></div>\n<ul>\n<li>To install OGRE to <code>/usr/local/lib/OGRE</code>:</li>\n</ul>\n<div><div id=\"highlighter_397549\" class=\"syntaxhighlighter href=\"https://codeyarns.com/2017/04/16/the-strange-case-of-the-notebook-that-woke-up-at-night/\" rel=\"bookmark\">The strange case of the notebook that woke up at&nbsp;night</a></h2>\t</header><!-- .entry-header -->\n\n\t<div class=\"entry-content\">\n\t\t<h1>Problem</h1>\n<p>In the last week, I started noticing that my notebook, that I had suspended before going to sleep would have woken up during the night! This notebook was running <strong>Ubuntu 16.04</strong>. It took a bit of investigation to realize that the notebook was being woken up by my new <strong>Targus AMW071</strong> wireless mouse. Some slight vibration to the desk during the night would move the mouse and it would wake up Linux from its sleep!</p>\n<h1>Solution</h1>\n<p>The list of devices that can wake up Linux from sleep is listed in the file <code>/proc/acpi/wakeup</code>. Each line of that file has a four-letter device code, obtained from BIOS, and whether that device is enabled (can wakeup from sleep) or disabled.</p>\n<p>I checked which devices could wakeup my Linux:</p>\n<div><div id=\"highlighter_292028\" class=\"syntaxhighlighter  plain\"><table border=\"0\" cellpadding=\"0\" cellspacing=\"0\"><tbody><tr><td class=\"gutter\"><div class=\"line number1 index0 alt2\">1</div><div class=\"line number2 index1 alt1\">2</div><div class=\"line number3 index2 alt2\">3</div><div class=\"line number4 index3 alt1\">4</div><div class=\"line number5 index4 alt2\">5</div></td><td class=\"code\"><div class=\"container\"><div class=\"line number1 index0 alt2\"><code class=\"plain plain\">$ cat /proc/acpi/wakeup | grep enabled</code></div><div class=\"line number2 index1 alt1\"><code class=\"plain plain\">XHC&nbsp;&nbsp; S3&nbsp;&nbsp;&nbsp; *enabled&nbsp;&nbsp; pci:0000:00:14.0</code></div><div class=\"line number3 index2 alt2\"><code class=\"plain plain\">PWRB&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; S3&nbsp;&nbsp;&nbsp; *enabled&nbsp;&nbsp; platform:PNP0C0C:00</code></div><div class=\"line number4 index3 alt1\"><code class=\"plain plain\">SLPB&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; S3&nbsp;&nbsp;&nbsp; *enabled&nbsp;&nbsp; platform:PNP0C0E:00</code></div><div class=\"line number5 index4 alt2\"><code class=\"plain plain\">LID0&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; S3&nbsp;&nbsp;&nbsp; *enabled&nbsp;&nbsp; platform:PNP0C0D:00</code></div></div></td></tr></tbody></table></div></div>\n<p>The culprit here was <code>XHC</code>, the USB 3.0 hub to which the mouse was connected. <code>PWRB</code> and <code>SLPB</code> are Power buttons and <code>LID0</code> is the notebook lid. So, I had to disable the <code>XHC</code> device in this file. Now, if the device name is written to this file, that operation toggles the enabled or disabled state for that device.</p>\n<p>So, to disable my mouse from waking up my notebook, I created this shell script:</p>\n<div><div id=\"highlighter_375205\" class=\"syntaxhighlighter  plain\"><table border=\"0\" cellpadding=\"0\" cellspacing=\"0\"><tbody><tr><td class=\"gutter\"><div class=\"line number1 index0 alt2\">1</div><div class=\"line number2 index1 alt1\">2</div><div class=\"line number3 index2 alt2\">3</div></td><td class=\"code\"><div class=\"container\"><div class=\"line number1 index0 alt2\"><code class=\"plain plain\">#!/bin/bash</code></div><div class=\"line number2 index1 alt1\"><code class=\"plain plain\"># disable_wakeup.sh</code></div><div class=\"line number3 index2 alt2\"><code class=\"plain plain\">echo \"XHC\" &gt; /proc/acpi/wakeup</code></div></div></td></tr></tbody></table></div></div>\n<p>And I executed that file with superuser privileges:</p>\n<div><div id=\"highlighter_449649\" class=\"syntaxhighlighter  plain\"><table border=\"0\" cellpadding=\"0\" cellspacing=\"0\"><tbody><tr><td class=\"gutter\"><div class=\"line number1 index0 alt2\">1</div></td><td class=\"code\"><div class=\"container\"><div class=\"line number1 index0 alt2\"><code class=\"plain plain\">$ sudo ./disable_wakeup.sh</code></div></div></td></tr></tbody></table></div></div>\n<p>I could see that this had disabled the mouse from waking up my notebook:</p>\n<div><div id=\"highlighter_530020\" class=\"syntaxhighlighter  plain\"><table border=\"0\" cellpadding=\"0\" cellspacing=\"0\"><tbody><tr><td class=\"gutter\"><div class=\"line number1 index0 alt2\">1</div><div class=\"line number2 index1 alt1\">2</div><div class=\"line number3 index2 alt2\">3</div><div class=\"line number4 index3 alt1\">4</div></td><td class=\"code\"><div class=\"container\"><div class=\"line number1 index0 alt2\"><code class=\"plain plain\">$ cat /proc/acpi/wakeup | grep enabled</code></div><div class=\"line number2 index1 alt1\"><code class=\"plain plain\">PWRB&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; S3&nbsp;&nbsp;&nbsp; *enabled&nbsp;&nbsp; platform:PNP0C0C:00</code></div><div class=\"line number3 index2 alt2\"><code class=\"plain plain\">SLPB&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; S3&nbsp;&nbsp;&nbsp; *enabled&nbsp;&nbsp; platform:PNP0C0E:00</code></div><div class=\"line number4 index3 alt1\"><code class=\"plain plain\">LID0&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; S3&nbsp;&nbsp;&nbsp; *enabled&nbsp;&nbsp; platform:PNP0C0D:00</code></div></div></td></tr></tbody></table></div></div>\n<p>I suspended the notebook and also verified that indeed the mouse could not wake it up.</p>\n<p>The only problem left now was that this toggle would apply only for this Linux session. A system reboot would put back the <code>XHC</code> device in enabled state. To run this toggle script on every system reboot:</p>\n<div><div id=\"highlighter_573693\" class=\"syntaxhighlighter  plain\"><table border=\"0\" cellpadding=\"0\" cellspacing=\"0\"><tbody><tr><td class=\"gutter\"><div class=\"line number1 index0 alt2\">1</div><div class=\"line number2 index1 alt1\">2</div></td><td class=\"code\"><div class=\"container\"><div class=\"line number1 index0 alt2\"><code class=\"plain plain\">$ sudo crontab -e</code></div><div class=\"line number2 index1 alt1\"><code class=\"plain plain\">@reboot /home/joe/scripts/disable_wakeup.sh</code></div></div></td></tr></tbody></table></div></div>\n<p>Now I had a permanent solution that makes sure that the mouse can never wake up my Ubuntu from sleep! The executable ran without problems. .entry-header -->\n\n\t<div class=\"entry-content\">\n\t\t<h1>Problem</h1>\n<p>If you try to <code>cp</code> or <code>mv</code> a large number of files or directories, you will encounter this error:</p>\n<div><div id=\"highlighter_287273\" class=\"syntaxhighlighter  plain\"><table border=\"0\" cellpadding=\"0\" cellspacing=\"0\"><tbody><tr><td class=\"gutter\"><div class=\"line number1 index0 alt2\">1</div><div class=\"line number2 index1 alt1\">2</div><div class=\"line number3 index2 alt2\">3</div><div class=\"line number4 index3 alt1\">4</div><div class=\"line number5 index4 alt2\">5</div></td><td class=\"code\"><div class=\"container\"><div class=\"line number1 index0 alt2\"><code class=\"plain plain\">$ cd big_dir</code></div><div class=\"line number2 index1 alt1\"><code class=\"plain plain\">$ cp * ../../some_other_dir</code></div><div class=\"line number3 index2 alt2\"><code class=\"plain plain\">Failed to execute process '/bin/cp'. Reason:</code></div><div class=\"line number4 index3 alt1\"><code class=\"plain plain\">The total size of the argument and environment lists 3.2MB exceeds the operating system limit of 2MB.</code></div><div class=\"line number5 index4 alt2\"><code class=\"plain plain\">Try running the command again with fewer arguments.</code></div></div></td></tr></tbody></table></div></div>\n<p>Note that you can copy or move one directory that contains a million files or any number of files for that matter. This error happens only if the shell has to pass a large number of input arguments to the <code>cp</code> or <code>mv</code> programs. So, if you run the above command from inside a directory containing 100K files, you will surely get this error.</p>\n<p>This is because there is a limit to the size of the arguments and environment strings that can be passed to a program. That limit can be queried:</p>\n<div><div id=\"highlighter_866032\" class=\"syntaxhighlighter  plain\"><table border=\"0\" cellpadding=\"0\" cellspacing=\"0\"><tbody><tr><td class=\"gutter\"><div class=\"line number1 index0 alt2\">1</div><div class=\"line number2 index1 alt1\">2</div></td><td class=\"code\"><div class=\"container\"><div class=\"line number1 index0 alt2\"><code class=\"plain plain\">$ getconf ARG_MAX</code></div><div class=\"line number2 index1 alt1\"><code class=\"plain plain\">2097152</code></div></div></td></tr></tbody></table></div></div>\n<p>The result will vary on different computers. But they always have a limit and it is encoded in the Linux kernel. There does not seem to be an userspace method to increase this size.</p>\n<h1>Solution</h1>\n<p>Instead of looking to increase the <code>ARG_MAX</code> size, examine the real problem. Why make the shell expand all the filenames and pass them as one gigantic list of strings to the programs? Instead these alternate solutions can be tried:</p>\n<ul>\n<li>See if you can instead <code>cp</code> or <code>mv</code> a parent directory, instead of a million files.<p></p>\n</li>\n<li>\n<p>Move the files one by one by writing a <strong>loop</strong> in shell script or Python.</p>\n</li>\n<li>\n<p>Use other programs like <strong>rsync</strong> to copy a directory to the destination.</p>\n</li>\n</ul>\n<p><strong>Reference:</strong> Search for <code>ARG_MAX</code> in the <a href=\"http://man7.org/linux/man-pages/man2/execve.2.html\">execve(2)</a> title=\"Configuring Bluetooth headphones\"></p>\n<p>One of the irritating problems with <strong>Linux</strong> in general is <strong>Bluetooth</strong>. Below are the steps I had to follow to pair and use a <strong>Creative WP-300</strong> Bluetooth headphones with <strong>Kubuntu 16.04</strong>. The procedure should be similar for any Bluetooth headphones or speaker and with any other variant of Ubuntu.</p>\n<ul>\n<li>Make sure that your computer has a Bluetooth adapter and it is working. Do not just assume there is an adapter and its working in Linux! You should be able to see a Bluetooth icon in the system panel or system tray. In the Bluetooth settings you should be able to see the adapter.<p></p>\n</li>\n<li>\n<p>Power on your Bluetooth headphones or speaker and put it into <strong>pairing mode</strong>. Refer to its documentation if you do not know how to do this.</p>\n</li>\n<li>\n<p>Go to the Bluetooth settings in Linux and try to <strong>find</strong> device and once the device is listed, <strong>pair</strong> with it and connect to it.</p>\n</li>\n<li>\n<p>Go to the audio settings and you should be able to see the Bluetooth device listed there. If it is not there, you might need to restart Linux. I know it is crazy, I had to do this, since no other solution offered online worked for me!</p>\n</li>\n<li>\n<p>If your device is listed in audio settings then try to play some music or video and see if it plays in the device. Most probably, it will still play on the default speaker of your computer! Go to System Settings → Multimedia → Audio and Video → Device Preference → Audio Playback. rel=\"bookmark\">Aligned memory allocation</a></h2>\t</header><!-- .entry-header -->\n\n\t<div class=\"entry-content\">\n\t\t<p>In some scenarios, you want to get memory that is <strong>aligned</strong> at an address that is a certain power of 2. Certain CPU architectures and certain operations require (or are faster) if their operands are located at an address that is a multiple of a certain power-of-2 number. For these reasons, you might see that many multi-platform libraries use an aligned memory allocator instead of <strong>malloc</strong> in their code. For example, OpenCV uses methods named <strong>fastMalloc</strong> and <strong>fastFree</strong> inside its code that do this type of allocation and freeing.</p>\n<p>Most of these methods work like this:</p>\n<ul>\n<li>They internally get memory from malloc. However, if you requested for N bytes, the wrapper will request for N+P+A bytes from malloc. Here, P is the size of a pointer on that CPU architecture and A is the alignment required, expressed in power-of-2 number of bytes. For example, if I request for 100 bytes on a 64-bit CPU and require the memory to be aligned to a multiple of 32, then the wrapper will request for 140 bytes.<p></p>\n</li>\n<li>\n<p>After getting the memory from malloc, it aligns the pointer forward so that (1) the pointer is at an address that is aligned as per requirement and (2) there is space behind the pointer to store a memory address.</p>\n</li>\n<li>\n<p>Then we sneak and store the address actually returned by malloc behind the pointer address and return the pointer to the user.</p>\n</li>\n<li>\n<p>The user has to use our free wrapper to free this pointer. When she does that we sneak back to reveal the actual address returned by malloc and free using that.</p>\n</li>\n</ul>\n<p>Here is some example code that illustrates aligned memory
