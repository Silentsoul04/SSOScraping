Version Manager (RVM)\n          </h1>\n          <p>\n            RVM is a command-line tool which allows you to easily install, manage, and work with multiple ruby\n            environments from interpreters to sets of gems.\n          </p>\n        </div>\n        <img alt=\"w00t!\" src=\"/images/rvm-logo-all-happy.png\" style=\"height: 204px\">\n        <h1 class=\"text-center\">\n RailsCasts.com on Getting Started with Rails\n            </a>\n          </li>\n        </ul>\n        <hr>\n        <h2>Community Maintained Project</h2>\n        <p>\n          RVM is maintained and supported by the community via issue trackers and pull requests:\n          </p><ul>\n            <li>\n              <a href=\"https://github.com/rvm\">\n                on each project in RVM Github organization,\n              </a>\n            </li>\n            <li>\n              especially\n              <a href=\"https://github.com/rvm/rvm/issues\">\n                RVM project issues tracker.\n              </a>\n            </li>\n          </ul>\n        <p></p>\n        <h2>Production</h2>\n        <p>\n          RVM  lets you deploy each project with its own completely self-contained and dedicated environment,\n          from the specific version of ruby, all the way down to the precise set of required gems to run your\n          application. Having a precise set of gems also avoids the issue of version conflicts between\n          projects, which can cause difficult-to-trace errors and hours of hair loss. With RVM, NO OTHER\n          GEMS than those required are installed. This makes working with multiple complex applications, where\n          each has a long list of gem dependencies, much more efficient. RVM lets you easily test gem upgrades,\n          by switching to a new clean set of gems to test with, while leaving your original set intact.\n          It is flexible enough to even let you maintain a set of gems per environment, or per development\n          branch, or even per individual developer's taste!\n        </p>\n        <h2>Development</h2>\n        <p>\n          RVM reduces the complexity of the many facets of ruby development through\n          its command-line API.  With RVM, you can have **identical** self-contained\n          environments in your Development, CI, Q/A, Staging, and Production environments.\n          No more hidden gotchas: if it works for one of them, it will work in all of them.\n          With named gemsets, you can reliably and immediately push changes from one environment to another.\n          And, RVM is for Ruby applications, *not just for Rails*!\n          <em>Any</em>\n          Ruby based application will benefit from your use of RVM.\n        </p>\n        <h2>Testing</h2>\n        <p>\n          RVM enables you to easily test both upgrade and escape paths very easily and consistently.\n          With RVM, you can run a test suite, rake tasks, benchmarks and gem commands against multiple ruby\n          versions\n          <em>at the same time.</em>\n          This means that you can easily ensure your applications work in Ruby 1.8.X, REE, MRI 1.9.1, jruby,\n          etc... and quickly expose any areas in which they do not.\n        </p>\n        <h2>Gem Management</h2>\n        <p>\n          RVM has an extremely flexible gem management system called Named Gem Sets. RVM's 'gemsets' make\n          managing gems across multiple versions of Ruby a non-issue. RVM lets you add a small text file to\n          your application's repository, instead of checking in tons of gems which would needlessly inflate\n          your repository size. Additionally, RVM's gemset management uses a common cache directory, so only\n          one downloaded version of each gem resides on disk rather than several copies.\n        </p>\n        <p>\n          RVM helps ensure that all aspects of Ruby are completely contained within user space, strongly\n          encouraging safer, non-root use. Use of RVM rubies thus provides a higher level of system security,\n          and therefore reduces risk and cuts overall system downtime. Additionally, since all processes run at\n          the user level, a compromised ruby process cannot compromise the entire system.\n        </p>\n        <h2>Support</h2>\n        <p>\n          RVM is continuously evolving, thanks to community pull requests to the project.\n        </p>\n        <a id=\"bottom\"></a>\n      </div>\n      <div id=\"docindex\">\n        <div class=\"headline\">\n
